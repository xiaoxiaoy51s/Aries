# TRAE Debugger - 运行时调试技能

## 适用场景

- 仅通过静态代码分析无法诊断的 Bug
- 偶发性、环境相关、或需要运行时证据的复杂问题
- 需要插桩日志、断点调试或采集运行时数据的场景

## 调试流程

采用科学调试方法论：**假设 → 插桩 → 复现 → 分析 → 修复 → 验证**

### 第 1 步：收集信息

- 理解 Bug 的表现、触发条件和复现步骤
- 检查相关日志、错误栈、监控指标
- 确认受影响的环境（开发/测试/生产）

### 第 2 步：提出假设

- 基于已有信息，提出一个或多个根因假设
- 每个假设都应可验证（可观测到特定证据）

### 第 3 步：插桩采集

- 在关键路径添加临时日志或断点
- 确保插桩不会改变程序行为（无副作用）

### 第 4 步：复现并验证

- 按复现步骤触发 Bug
- 收集插桩输出的运行时证据
- 验证或排除假设

### 第 5 步：定位根因

- 确认问题根因
- 评估影响范围

### 第 6 步：修复并验证

- 实施修复
- 验证 Bug 不再复现
- 清理临时插桩代码

## 调试技巧

| 问题类型 | 推荐工具/方法 |
|----------|-------------|
| 逻辑错误 | 添加关键变量日志，分步对比预期值 |
| 并发问题 | 记录线程/协程 ID、锁状态、竞态条件 |
| 内存问题 | 检查对象引用、缓存大小、内存泄漏模式 |
| 网络问题 | 抓包、检查超时配置、DNS 解析 |
| 性能问题 | 耗时打点、火焰图、慢查询日志 |

## 输出格式

```markdown
## 调试报告

### 问题描述
[Bug 的复现步骤和表现]

### 分析过程
1. **假设 H1**：[假设内容]
   - 验证：[验证方法和结果]
   - 结论：[成立/排除]
2. **假设 H2**：[...]

### 根因
[确定的根因]

### 修复方案
[具体修复代码或配置变更]

### 验证结果
[修复后的验证结论]
```
