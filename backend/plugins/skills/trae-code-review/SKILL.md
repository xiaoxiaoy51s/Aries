# TRAE Code Review - 代码审查技能

## 适用场景

审查合并请求（PR/MR）、代码差异（diff）、或任意代码片段的结构化审查。

## 审查流程

### 1. 理解变更上下文

- 阅读相关代码文件，理解变更的目的和范围
- 检查是否有相关的 issue、需求文档或设计文档

### 2. 逐项审查

从以下维度逐项检查代码：

| 维度 | 检查要点 |
|------|----------|
| 正确性 | 逻辑是否正确？边界条件是否处理？是否存在并发问题？ |
| 可读性 | 命名是否清晰？代码是否自文档化？复杂度是否合理？ |
| 性能 | 是否存在性能瓶颈？是否有效率低下的算法或查询？ |
| 安全性 | 是否存在注入风险？输入是否经过验证？敏感信息是否泄露？ |
| 可维护性 | 是否遵循项目约定？是否有重复代码？测试是否充分？ |
| 兼容性 | API 变更是否向后兼容？数据库迁移是否安全？ |

### 3. 输出结构化反馈

按严重程度分级：

- **Critical**：可能导致生产事故或数据丢失的问题
- **Major**：明显的逻辑错误或严重违反最佳实践
- **Minor**：可改进的小问题，不影响功能
- **Suggestion**：改进建议，非强制要求

## 输出格式

```markdown
## 审查结果

### 总体评价
[一句话总结代码质量]

### Issues

#### Critical/Major
- **文件:行号** - 问题描述
  - 建议修复：[具体方案]

#### Minor
- **文件:行号** - 问题描述

#### Suggestions
- **文件:行号** - 改进建议

### 好的实践
- 值得肯定的代码写法

### 总结
[整体评估和建议]
```
