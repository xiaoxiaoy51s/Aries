## 二、核心问题

### 问题 1：`utils/` 是万能杂物间

28 个文件，从 40KB 的执行引擎到 245B 的工具函数塞在一起，职能跨度大：

| 类别 | 文件 | 大小 | 应有归宿 |
|------|------|------|----------|
| **Agent 执行引擎** | `subagent_runtime.py` | 40KB | `engine/` |
| **Agent 配置加载** | `subagent_manager.py` | 17.6KB | `engine/` |
| **Agent 工具定义** | `agent_tools.py` | 16.2KB | `engine/` |
| **文件编辑工具** | `edit_tools.py` | 17.9KB | `engine/` |
| **技能管理** | `skills_manager.py` | 13KB | `engine/` |
| **MCP 运行时** | `mcp_runtime.py` | 26.9KB | `mcp/` |
| **MCP 配置** | `mcp_config.py` | 11.8KB | `mcp/` |
| **文件 IO 工具** | `file_manager.py` | 39.5KB | 可拆入 `engine/` |
| **平台渠道** | `feishu_link.py` / `wechat_link.py` | 11.7KB / 5KB | `services/` |
| **基础设施** | `session_logger.py` / `scheduler.py` / `app_setting.py` 等 | 4~12KB | 保留 `utils/` |

### 问题 2：大文件需要拆分（>500行 / >15KB）

| 文件 | 大小 | 行数 | 拆分建议 |
|------|------|------|----------|
| `utils/subagent_runtime.py` | 40.3KB | ~995 | → 4 文件（详见第四节） |
| `utils/file_manager.py` | 39.5KB | ~800 | → 2-3 文件 |
| `utils/mcp_runtime.py` | 26.9KB | ~550 | → 2 文件 |
| `api/modes/stream_handler.py` | 25.1KB | ~509 | → 2 文件 |
| `prompt/edit_code_prompts.py` | 23.0KB | ~500 | 提示词数据，可接受 |
| `api/git_api.py` | 19.4KB | ~400 | 可接受，可考虑拆响应模型 |
| `api/chat/chat.py` | 17.5KB | ~350 | 适中 |
| `utils/edit_tools.py` | 17.9KB | ~400 | → 归入 `engine/` |
| `utils/plugin_manager.py` | 17.6KB | ~400 | → 归入 `engine/` |
| `utils/subagent_manager.py` | 17.6KB | ~400 | → 归入 `engine/` |
| `api/files_api.py` | 17.7KB | ~400 | 可考虑拆分 API + 业务逻辑 |

### 问题 3：命名与职责不对齐

- `api/modes/` → 实际是 Agent 执行核心引擎，`modes` 命名太泛且表意不明
- `api/` 层的路由文件里混入了纯数据类/工具函数（如 `git_api.py` 里混了数据模型）
- `tools/` 目录空置，没有按原意图放工具实现
- `cli/` 目录完全为空

### 问题 4：入口文件的大 import

`main.py` 第 24 行单行从 `api` 导入了 **22 个 router**，违反单行可读性，也不好维护。

## 四、大文件拆分方案（详细）

### 4.1 `utils/subagent_runtime.py`（40KB / ~995行）→ 4 文件

当前文件内包含 6 个独立的职责：
1. **数据模型**：`SubagentExecution` dataclass + 全局取消注册表（~70行）
2. **工具定义构建**：`_build_skill_tool_definitions`, `_build_core_tool_definitions`, `_filter_mcp_tools`, `_execute_skill_tool`（~100行）
3. **Prompt 构建**：`_build_subagent_system_prompt`（~60行）
4. **凭证解析**：`_resolve_model_credentials`（~20行）
5. **编排函数**：`run_subagent` 主入口（~200行）
6. **执行循环**：`_run_subagent_loop`（~400行）—— 包含完整的 SSE 流式解析、工具调用、事件推送

**拆分方案**：

```
engine/
├── subagent_runtime.py       run_subagent() 编排 + _run_subagent_loop() 循环（精简后 ~500行）
├── subagent_types.py         数据模型：SubagentExecution + cancel 注册表（~100行）
├── subagent_tools.py         工具构建：_build_skill_tool_definitions 等（~150行）
└── subagent_prompt.py        Prompt 构建 + 凭证解析（~100行）
```

### 4.2 `utils/file_manager.py`（39.5KB / ~800行）→ 2-3 文件

这个文件集成了文件读取、写入、搜索、遍历等所有文件操作。建议按操作类型拆分：

```
engine/
├── file_manager.py           保留：read_file / write_file / delete_file / list_files
└── file_search.py            迁出：search_file（ripgrep 封装层 ~200行）
```

（注：`edit_tools.py` 是另一套文件编辑工具，保持独立）

### 4.3 `api/modes/stream_handler.py`（25.1KB / ~509行）→ 拆分 or 重命名

这个文件包含主 Agent 的多轮工具调用循环，是整个系统的核心。

**选项 A（推荐）**：将核心循环逻辑迁入 `engine/main_agent.py`，`api/modes/` 只保留 SSE 流式转换。

**选项 B（最小改动）**：将文件拆为：
```
api/modes/
├── stream_handler.py     保留：流式循环编排 + SSE 事件推送（~300行）
├── stream_logic.py       迁出：_stream_once / _process_tool_calls / _check_repeat（~200行）
└── stream_constants.py   保留不变
```

### 4.4 `utils/mcp_runtime.py`（26.9KB / ~550行）→ 迁入 `mcp/`

```
mcp/
├── runtime.py              MCP 连接池 + 工具执行 + get_mcp_tool_definitions
├── config.py               配置加载（从 utils/mcp_config 合并）
└── definitions.py          工具定义构建（从 runtime.py 拆出 ~100行）
```

## 五、操作路径

### 第一阶段：低成本快速重组（不改 import，不改功能）

1. **重命名 `api/modes/` → `api/engine/`**（纯目录重命名，涉及 import 约 5 处）
2. **删除空目录 `tools/` 和 `cli/`**
3. **拆分 `utils/subagent_runtime.py` 为 4 文件**（在 `utils/` 内新建文件，保留原文件做兼容 import）

### 第二阶段：目录迁移（需改 import）

4. **创建 `engine/` 包**，从 `utils/` 迁入子 Agent 相关文件
5. **创建 `mcp/` 包**，从 `utils/` 迁入 MCP 相关文件
6. **将 `feishu_link.py` 和 `wechat_link.py` 合并进 `services/`**

### 第三阶段：最终清理

7. 整理 `main.py` 的 import 为多行
8. 确保 `utils/` 只保留基础设施工具

| 指标 | 当前 | 重组后 |
|------|------|--------|
| `utils/` 文件数 | 28 | ~15 |
| >15KB 的文件数 | 12 | 6-7 |
| 目录层级清晰度 | 混乱（utils 什么都有） | 清晰（engine/mcp/services 各司其职） |

| 循环引用风险 | 中 | 低 |
